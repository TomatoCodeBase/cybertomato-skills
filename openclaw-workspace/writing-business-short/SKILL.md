# 商业短文写作

50-500字的口语化短文，活人感，IP化。

## 执行流程（收到写作请求后的标准步骤）

```
1. 确认主题 → 检查是否触碰微信红线（AI自动化写作等），踩红线直接拒绝
2. 确认弹药类型 → 至少选定一种可收藏元素（对比表/结构示例/命令步骤/真实数据/工作流）
3. 选择技法组合 → 根据文章类型匹配（见下方技法选择指南）
4. 输出大纲 → ⏸ 展示给用户确认方向和素材来源
5. 按大纲+技法写正文 → 确保每个示例来自作者真实经历
6. 16项检查 → 连续2轮无明显可优化才算通过
7. 生成爆款标题 → 调用writing-title-wechat
8. 保存到本地vault + 告知用户
```

**⏸ 大纲确认点不可跳过**——防止写完因方向错误而整篇作废。

### 技法选择指南

| 文章类型 | 必选技法 | 推荐技法 |
|----------|---------|---------|
| 工具/教程类 | 技法0(信息增量)+技法2(编号)+技法2.5(升级) | 技法4(预判疑问)+技法5(比喻) |
| 副业/踩坑类 | 技法1(白描开局)+技法3(论断支点) | 技法6(金句)+技法7(跳笔) |
| 观点/认知类 | 技法1(白描开局)+技法6(金句) | 技法8(10%法则)+技法3(论断支点) |
| 对比/选型类 | 技法0(信息增量)+技法2.5(升级) | 技法3(论断支点)+技法5(比喻) |

## 核心原则

**基于事实，可落地可复现。**

写出来的每一个字，读者都要能验证、能照做、能复现。写自己真实做过的事，写读者真的能跑通的命令，写经过验证的效果。

基于事实 = 不编数据、不搬别人的案例、不写自己没跑通过的命令
可落地 = 每条建议都有具体操作步骤
可复现 = 读者照着做能跑出一样的结果

**普适性洞见 > 情境性观点**
**思维模式 > 具体结论**
**可迁移智慧 > 专属经验**

用最少的字，把事说明白且有价

## 写作原则

### 内容
- 聚焦一个具体问题，针对性解答
- 横向知识汇总：看完不用再翻其他文章
- 利他：别人看了能获得价值
- 聚焦特定人群

### 风格（从实战文章提炼）

> 以下风格特征来自「AI助手自我进化」等已发布文章的风格分析器提炼。
> 原则：普适性洞见 > 情境性观点；思维模式 > 具体结论；可迁移智慧 > 专属经验

**句式**：
- 短句独占行（3-8字高频），单句≤15字
- 设问自答推进，对比句式（A vs B）
- 关键句独占一行，段间空行留白

**用词**：
- 正式度1/5，纯口语
- 高频特征词：说白了、区别在哪、扎心、顿悟、一句话、关键是
- 规避：总而言之、不可否认、应该、必须（说教类）

**修辞**：
- 白描（纯场景，无形容词堆砌）
- 生活化比喻（止疼药=治标、实习生=AI人格化、螺旋=进化可视化）
- 重复动词对比（不在"做不做"，在"想不想"）

**叙事**：
- 第一人称亲历者视角，自我剖析式（承认错误比展示成果更用力）
- 倒叙开局（先结果后过程），关键转折精确到时间点
- 冷静中带自嘲，理性中有人味，情感强度2/5

**思维**：
- 推进路径：现象→追问根因→提炼模式→给出方法→抽象普适洞见
- 深度4/5，快慢交替节奏（白描段快，论断段慢）
- 金句提炼：用强动词开头 + 重复动词前后对比

**节奏**：
- 快慢交替：白描段快节奏跳切，论断段慢下来独占行
- 段落长度：手机适配，每段≤100字（约5行），关键句独占行
- 不用过渡句，用空行和跳切代替

### SEO
- 包含目标用户能搜到的关键词

### 字数
- 50-500字（灵活，根据内容定）
- 底线：不低于50字
- 核心：能短就短

## 写作技法（从实战提炼）

> 以下技法来自陈铭《写作28讲》+ 番茄文章优化实践，每次写作必须遵循。

### 技法0：信息增量优先

技法是为信息服务的。如果一篇文章技法满分但读者学不到新东西，等于零。

工具类/教程类文章，必须包含：
- **完整技术栈**：用了哪些工具/技能，一个不落
- **可复现步骤**：读者照着做能跑通
- **安装命令/配置方法**：不是概述，是真实命令

```
❌ 帮我安装 zhihu-auto skill（这是作者自建技能，公开仓库不存在，读者照做必报错）
✅ 帮我安装 web-access skill，仓库地址是 https://github.com/eze-is/web-access
   （真实存在的仓库，读者照做能跑通）
```

**原理**：读者搜索文章是为了解决问题，不是为了欣赏文笔。信息不全，文笔再好也是空壳。

**小白复现测试**：写完每条命令前，问自己——如果我是第一次看到这篇文章的人，照着做能跑通吗？跑不通就不能写。

---

### 技法1：白描开局

文章开头**不用论断/观点**，用具体场景。谁+在哪+干什么。让读者3秒代入。

```
❌ AI最擅长的事，是你最懒得做的事。（论断开局，读者还没代入就收到判断）
✅ 你有没有这种经历？设了三个闹钟提醒自己打卡。
   早上醒来看一眼——知乎创作分掉了，头条活动断了。（场景开局，读者看见自己）
```

**原理**：读者对场景的共鸣力 > 对论断的认同感。先让人看见自己，再下判断。

### 技法2：教程编号化

涉及操作步骤，必须编号（第一步/第二步/第三步），每步一句话。

```
❌ AI直接连Chrome浏览器，天然带登录态。你只要装个skill，把链接发过去，
   跟它说一句话就行……（连续段落，读者自己拆解）
✅ 怎么搭？三步。
   第一步，安装web-access skill。
   第二步，把知乎链接发给OpenClaw。
   第三步，说一句话。
   （编号拆分，认知负担最低）
```

**原理**：手机阅读时代，读者的耐心只够你一步一步带。连续段落 = 流失。

### 技法2.5：教程升级法则——教程是骨架，决策是血肉

教程类文章初稿容易写成纯步骤清单（第一步干嘛、第二步干嘛）。但番茄的真实修改规律表明：**读者要的不是说明书，是一个值得跟着走的人。**

**核心规律：把"怎么做"升级为"为什么做"。**

教程告诉读者怎么做，决策告诉读者为什么做、值不值得做、别人怎么选的。角度一变，整篇文章的势能就变了。

**升级三要素**：

1. **加决策逻辑**——公开"我为什么做这个选择"，权衡利弊的过程比步骤本身更有说服力。教程型：第一步申请API→第二步安装→第三步选系统。决策型：为什么换？（沉没成本不是成本）→换的过程中我做了什么取舍？→结果怎么样？
2. **加刹车信息**——对比表/推荐不只给油门（选它+为什么好），还要给刹车（使用前要想清楚什么）。3列表（你想干嘛/选它/为什么）是帮人做选择，5列表（系统/适合什么/像什么/优势/要想清楚）是帮人做决策。一列给类比直觉，一列给风险提醒。
3. **加社会证明**——社区数据、群里多少人、自费多少升级——行动信号告诉读者"不只我一个人在做这件事"。

**金句也要跟着升级**：

- 教程级金句停在操作层：装Hermes不难，难的是没人告诉你先后顺序
- 决策级金句跳到认知层：Skill就是工作流，是个人和组织knowhow的沉淀

```
❌ 教程型（纯步骤）：
   第一步：申请API
   第二步：建文件夹
   第三步：安装
   第四步：选记忆系统
   第五步：迁移数据
   （读者：知道了，但该不该换？）

✅ 决策型（教程+决策+社会证明）：
   一、为什么换？（决策逻辑+沉没成本）
   二、7个记忆系统怎么选？（5列表：系统/适合什么/像什么/优势/要想清楚）
   三、迁移实战（保留教程骨架，嵌入个人判断"我没全搬，只搬了技能"）
   四、社区热度（550人飞书群、自费600升级）
   （读者：有人在前面走，我也敢跟）
```

**实战来源**：Hermes五步安装文章。AI初稿是纯5步教程，番茄发布版改为"断舍离OpenClaw，全面切换Hermes Agent"——核心差异不是改几个字，是整体角度从"怎么做"变成了"为什么做"。

### 技法3：论断需支点

每个关键论断至少给2-3个支撑（数据/类比/案例）。裸论断 = 不可信。

```
❌ 把重复的事交给AI，人的精力才能花在真正需要思考的地方。（只有论断，没有支撑）
✅ 每天省下10分钟打卡，一个月就是5个小时。5个小时够写两篇稿子了。（数据支点）
```

**原理**：三角形是最稳固的结构。三个支点让论断站稳。一个支点容易被推翻，零个就是空中楼阁。

### 技法4：预判疑问

教程类文章，至少预判1个读者疑问并主动回答。制造节奏变化，消除信任障碍。

```
❌ AI直接控制你的Chrome浏览器。它带着你的登录态，到点自动检查。
   （平铺直叙，读者内心在问号）
✅ AI直接控制你的Chrome浏览器。
   你可能在想：这事靠谱吗？会不会乱发东西？
   我用了两周，没出过一次事故。（主动回答，信任升级）
```

**原理**：读者不会提问，只会关掉文章。你替他问出来，他反而觉得你懂他。

### 技法5：比喻记忆点

效果展示或转折段，用一个具体比喻收尾。让读者带走一个画面。

```
❌ 清明三天假期，知乎连打三天，头条也没断。两个平台全绿。（数据陈述，看完即忘）
✅ 清明三天假，我一天没亲自去打卡。两边全绿。
   这感觉怎么说呢——像有人帮你把早饭做好了，你只管吃。（比喻收尾，记住画面）
```

**原理**：人记住的不是数据，是画面。一个精准的比喻 = 一句免费的金句。

### 技法6：金句提炼

核心论断必须提炼为可传播、可记忆的金句。两个技巧：

**技巧1：用强动词开头**

```
❌ AI的主动性很重要。（平淡，无记忆点）
✅ 杀死孩子的大脑，你只需要在他玩玩具的时候不断打断他。（"杀死"强动词开头，瞬间抓人）
✅ 螺旋上升，不是原地打转。（动词对比，节奏感强）
```

**技巧2：重复同一个动词做前后对比**

```
❌ 做和不做有区别。（无力度）
✅ 不在"做不做"，在"想不想"。（重复动词+对比，洗脑感）
✅ 提示词教AI说话。规则文件教AI思考。（重复结构+对比，一句话带走核心观点）
```

**原理**：读者一周后不记得文章内容，只记得激动的感觉和一两句金句。金句是文章的传播锚点。

### 技法7：跳笔切分

长段切碎，大量分段，几乎不用过渡句。让读者从任何段落捡起来都能接着读。

```
❌ AI在后台跑完任务，默默把文件写好了。对我来说——啥也没看到。（连续段落，无呼吸感）
✅ AI在后台跑完任务，默默把文件写好了。对我来说——

   啥也没看到。（跳切+留白，读者停顿，画面感更强）
```

**原理**：手机阅读时代，长段落=流失。不到100字就分段。关键转折处用空行制造停顿，就像电影里的留白镜头。

### 技法8：10%法则

文章的信息和思考，只比读者聪明10%就够了。

```
❌ 标签索引器扫描183个记忆文件，生成657个标签，评分算法：精确×10+部分×3+标题×8+摘要×2
   （内部数据，读者不关心，信息过载）
✅ 一晚上，AI自己写了3条规则、2个脚本、7条教训。
   （读者能感知的结果，刚刚好）
```

**原理**：太浅=没增量，太深=烧脑流失。只比读者多走一步，让他觉得"跳一跳够得着"。内部数据（文件数、标签数、算法细节）对读者是噪音，砍掉。

## 检查流程

内容写完后，必须检查：

0. **信息增量**：技术栈完整吗？安装步骤可复现吗？读者能照着做吗？
1. **白描开局**：开头是具体场景还是抽象论断？（必须是场景，无论断）
2. **教程编号**：有操作步骤吗？编号了吗？每步一句话吗？
3. **教程升级**：教程类文章，有没有从"怎么做"升级到"为什么做"？决策逻辑/刹车信息/社会证明至少有一个？（见技法2.5）
4. **论断支点**：每个关键论断有2-3个支撑吗？
5. **预判疑问**：教程类文章，预判了读者疑问吗？
6. **比喻记忆点**：效果展示段有比喻收尾吗？
7. **金句提炼**：核心论断是否提炼为金句？金句在认知层还是操作层？（决策型文章需跳到认知层）
8. **跳笔切分**：段落≤100字？关键句独占行？手机适配？
9. **10%法则**：有无内部数据/术语过载？砍到读者能感知的粒度
10. **普适性检验**：情境性描述是否抽象为可迁移洞见？
11. **逻辑结构**是否清晰
12. **技术概念**是否通俗易懂，是否有案例
13. **语言**是否流畅，有无错别字或语病
14. **标题自检**（见下方清单）
15. **示例真实化**（见下方清单）

⚠️ 连续2轮检查后，认为没有明显可优化的地方，才算完成。

## 标题自检清单

- [ ] 是否包含痛点钩子？（"你有没有这种经历"、"XX混乱"）
- [ ] 是否给出解决方案？（"学会XX"、"用XX"）
- [ ] 是否承诺效果？（"效率翻倍"、"一句话搞定"）

**标题公式**：痛点 + 解决方案 + 效果承诺

```
❌ 工具名 + 功能描述
   → Get笔记+Obsidian+OpenClaw联动，一句话保存好文章

✅ 痛点 + 解决方案 + 效果
   → 知识管理混乱？学会OpenClaw+Obsidian+Get笔记，效率立马翻倍
```

## ⚠️ 命令/链接真实性检查（新增）

- [ ] 文章中每一条命令是否真实可执行？
- [ ] 每一个链接是否真实可访问？
- [ ] 安装步骤是否经过验证？

**原理**：一条假命令毁全文。读者照着执行报错，信任瞬间崩塌。

```
❌ npx skills add larksuite/cli --all -y -g（OpenClaw不支持此语法，读者执行会报错）
✅ 一句话："帮我装一下所有的东西：https://github.com/larksuite/cli/blob/main/README.zh.md"
```

---

## ⚠️ 对比类文章聚焦检查（新增）

写"X vs Y"或"X + Y"类文章时，后半段容易跑题：

- [ ] 文章是否和标题锁死？标题说X+Y，全文就只讲X+Y
- [ ] 后半段有没有偷偷变成Z的推介？

**原理**：读者耐心只够你讲一件事。想讲第二件，写第二篇。

```
❌ 标题：ClaudeCode + 飞书CLI → 后半段花500字讲OpenClaw怎么补救
✅ 标题：ClaudeCode + 飞书CLI → 全文围绕ClaudeCode + 飞书CLI，收束干净
```

---

## ⚠️ 场景归属检查（新增）

- [ ] 文中的场景是否来自作者真实经历？
- [ ] 是否直接搬运了别人的案例？

**原理**：别人的场景是素材，你的场景才是灵魂。用别人的场景写，读者会觉得"这跟我有什么关系"。

```
❌ "给24个同事批量发通知"（这是别人的体验，直接搬过来）
✅ "知识星球帖子自动采集到飞书表格"（这是作者自己的业务场景）
```

## 示例真实化检查

- [ ] 所有示例链接是否真实可点击？
- [ ] 占位符（xxx、TODO、https://xxx.com）是否已替换？
- [ ] 数据是否真实？（不用"几倍"、"大量"等模糊表达）

**原理**：真实案例建立可信度，占位符削弱说服力

```
❌ https://xxx.com/article（读者：假的吧？）
✅ https://mp.weixin.qq.com/s/g4Eo6csNItXPBd_w_5_lqA（读者：原来真的能用）
```

## 工具类文章结构（适用时）

如果是介绍工具/方法，按此结构：

1. **痛点引入**（场景共鸣，第一人称）
2. **决策逻辑**（为什么选它？权衡了什么？——技法2.5的核心落地位置）
3. **安装/配置**（两种方式：小白+进阶）
4. **使用示例**（真实可操作，**必须是作者自己的场景**）
5. **对比/选型**（如有竞品，给5列表：系统/适合什么/像什么/优势/要想清楚）
6. **社会证明**（社区数据/使用人数/个人投入——行动信号）
7. **本质洞见**（一句话点透底层逻辑，金句跳到认知层）
8. **一句话总结**

**核心写法**：
- 写"我用它干了什么"，不写"它能干什么"
- 体验感 > 介绍感。读者记住故事，记不住说明书
- 观点要原创：同一个观点，用自己的话重新说一遍，加上自己的思考
- **教程只是骨架，决策才是血肉**——纯步骤清单是初稿，加上"我为什么这么做"才是成品

**安装说明原则**：
- 一键入口（小白友好）
- 手动入口（进阶用户）
- 关键步骤明确说明（扫码授权/API密钥/配置文件）
- ⚠️ 每条命令必须验证可执行
