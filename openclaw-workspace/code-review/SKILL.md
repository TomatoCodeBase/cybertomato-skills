# code-review — 三维度并行代码审查

自动获取代码变更，并行执行代码复用、代码质量、效率三个维度的审查，汇总发现并直接修复。

## 触发词

`code review`、`代码审查`、`审查代码`、`simplify`

## 流程

### 第一步：获取变更

```bash
cd <project_path>
git diff                # 未暂存变更
git diff --cached       # 已暂存变更
git diff HEAD~1         # 最近一次提交
```

优先使用未暂存变更；若无变更则使用 `HEAD~1`。如果用户指定了文件路径或 commit range，按指定范围获取。

将 diff 输出保存到临时文件供子 agent 读取：
```
git diff > /tmp/code-review-diff.txt
```

### 第二步：并行三维度审查

使用 `sessions_spawn` 同时启动 3 个子 agent，每个负责一个维度。每个子 agent 接收 diff 内容和项目路径。

#### Agent A — 代码复用审查

检查项：
- 新代码是否重复了项目中已有的工具函数、辅助方法、常量
- 是否有现成的 npm 包 / 库函数可替代手写实现
- 相似逻辑是否可抽象为共享函数
- 是否有设计模式（factory、strategy、builder 等）能消除重复
- 已有测试工具、mock helper 是否被复用

输出格式：
```
## 复用问题
1. [文件:行号] 描述 → 建议复用: 已有函数/包
```

#### Agent B — 代码质量审查

检查项：
- 冗余状态：不必要的中间变量、过期的缓存、无用的 import
- 参数膨胀：函数参数 > 4 个时应考虑对象参数
- 复制粘贴：相似代码块 > 3 行应提取函数
- 字符串类型：应使用 enum / const 对象替代魔法字符串
- 不必要注释：解释"是什么"而非"为什么"的注释应删除
- 命名质量：模糊命名（data、info、temp、item）应替换为语义化名称
- 死代码：不可达分支、未使用的变量/函数/导出
- 类型安全：any 类型、缺失类型注解、不安全的类型断言

输出格式：
```
## 质量问题
1. [文件:行号] 描述 → 修复建议
```

#### Agent C — 效率审查

检查项：
- 重复计算：循环内不变的表达式应提到循环外
- 顺序执行可并行：无依赖关系的 async 操作应用 `Promise.all`
- 热路径膨胀：频繁调用的函数内不应有重型初始化或 console.log
- 内存泄漏：未清理的事件监听器、定时器、闭包引用
- 不必要的克隆：深拷贝只需浅拷贝的地方
- I/O 批量化：多次小读写应合并
- 过度渲染：React 组件不必要的 re-render（缺少 memo、useMemo）
- 懒加载缺失：大型模块应动态 import

输出格式：
```
## 效率问题
1. [文件:行号] 描述 → 优化建议
```

### 第三步：汇总与修复

收集 3 个 agent 的审查结果，按严重程度排序（🔴 必须修复 > 🟡 建议修复 > 🟢 可选优化）。

对 🔴 和 🟡 问题，使用 `edit` 工具直接修复。修复前确认修改范围，避免破坏现有功能。

### 第四步：输出摘要

```
## Code Review 完成

**审查范围**: <git 范围或文件>
**发现问题**: X 个
- 🔴 必须修复: A 个（已自动修复 B 个）
- 🟡 建议修复: C 个（已自动修复 D 个）
- 🟢 可选优化: E 个

### 修复摘要
1. [文件] 修复内容简述
2. ...

### 待手动处理
1. [文件:行号] 需要确认的变更描述
```

## 无子 Agent 时的降级方案

如果不具备 `sessions_spawn` 能力，串行执行三个维度的审查，每个维度聚焦最关键的 3-5 个问题，避免上下文膨胀。

## 注意事项

- 审查前先用 `exec` 确认项目语言和框架（检查 package.json、tsconfig.json、go.mod 等）
- 不修改测试文件中的断言逻辑，除非发现 bug
- 不删除看似无用但可能被反射/动态引用的代码
- 涉及配置文件、数据库 schema 的变更需额外谨慎，标记为 ⚠️ 需人工确认
- 对大规模重构建议（如架构调整、依赖替换）不自动执行，仅在摘要中列出
